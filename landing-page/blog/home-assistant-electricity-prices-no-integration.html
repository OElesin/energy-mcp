Home Assistant Electricity Prices Without a Custom Integration

Every Home Assistant user on a dynamic electricity tariff eventually needs the same thing: real-time spot prices as a sensor. And every one of them hits the same problems.

The Problem with Existing Integrations

The common options — Nordpool, ENTSO-E, EPEX Spot via HACS — all share the same architectural weakness: they run inside your Home Assistant Python process.

⚠️ Common complaints: "ENTSO-E completely hangs the system — no web UI, had to SSH in." / "Nordpool entities frequently become unavailable." / "Every HA supervisor update breaks Nordpool until I restart."

When these integrations crash, they can take your entire smart home with them. And because they're community-maintained HACS plugins, there's no SLA on fixes.

Integration Runs in HA? Can crash HA? Cheapest window? Cost estimate?

The Approach: External API, Built-in Sensor

Energy MCP runs on AWS Lambda, completely outside your Home Assistant instance. You query it using HA's built-in command_line sensor — no HACS, no custom component, no Python dependencies.

If our server has an issue, your sensor just shows unavailable. Your HA instance stays up. Your automations keep running.

Sensor 1: Current Electricity Price Stats

Get the average, minimum, and maximum spot price for today, plus which hours are cheapest and most expensive:

command_line:
  - sensor:
      name: electricity_price_stats
      command: >
        curl -s -X POST https://energy-mcp.getbrechtai.com/mcp
        -H "Content-Type: application/json"
        -H "Accept: application/json, text/event-stream"
        -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"get_price_stats","arguments":{}}}'
        | grep "^data:" | sed 's/^data: //'
      value_template: >
        {{ (value_json.result.content[0].text | from_json).stats.avg_ct_kwh | round(2) }}
      unit_of_measurement: "ct/kWh"
      scan_interval: 900

This gives you a sensor with the current average price as its state. Every 15 minutes it refreshes.

Sensor 2: Cheapest EV Charging Window

This is the one everyone wants. It finds the optimal consecutive hours to charge, and tells you the cost:

command_line:
  - sensor:
      name: cheapest_ev_window
      command: >
        curl -s -X POST https://energy-mcp.getbrechtai.com/mcp
        -H "Content-Type: application/json"
        -H "Accept: application/json, text/event-stream"
        -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"find_cheapest_window","arguments":{"duration_hours":3,"consumption_kwh":11}}}'
        | grep "^data:" | sed 's/^data: //'
      value_template: >
        {{ (value_json.result.content[0].text | from_json).start }}
      scan_interval: 1800

The sensor state is the optimal start time (e.g., 2026-07-20T02:00:00+02:00). Attributes include avg_price_ct_kwh, total_cost_eur, and end time.

Customize it: Change duration_hours and consumption_kwh to match your appliance. EV: 3h/11kWh. Heat pump: 4h/8kWh. Dishwasher: 2h/1.5kWh.

Sensor 3: Grid Carbon Intensity

Want to know if the grid is running clean right now? This sensor shows grams of CO₂ per kWh:

command_line:
  - sensor:
      name: grid_carbon_intensity
      command: >
        curl -s -X POST https://energy-mcp.getbrechtai.com/mcp
        -H "Content-Type: application/json"
        -H "Accept: application/json, text/event-stream"
        -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"get_carbon_intensity","arguments":{"country":"DE"}}}'
        | grep "^data:" | sed 's/^data: //'
      value_template: >
        {{ (value_json.result.content[0].text | from_json).carbon_intensity_gco2_kwh | round(0) }}
      unit_of_measurement: "gCO₂/kWh"
      scan_interval: 1800

Use this to run flexible loads during clean hours — when wind and solar are high, carbon intensity drops below 200 gCO₂/kWh.

Automation: Start Charging at the Cheapest Time

Combine the cheapest window sensor with a switch to automate your wallbox:

automation:
  - alias: "Start EV charging at cheapest window"
    trigger:
      - platform: template
        value_template: >
          {{ now().isoformat()[:16] ==
             states('sensor.cheapest_ev_window')[:16] }}
    action:
      - service: switch.turn_on
        target:
          entity_id: switch.ev_charger
      - service: notify.mobile_app
        data:
          title: "⚡ EV charging started"
          message: >
            Charging now — estimated cost: €{{ state_attr('sensor.cheapest_ev_window', 'total_cost_eur') }}

  - alias: "Stop EV charging after window"
    trigger:
      - platform: template
        value_template: >
          {{ now().isoformat()[:16] ==
             (state_attr('sensor.cheapest_ev_window', 'end'))[:16] }}
    action:
      - service: switch.turn_off
        target:
          entity_id: switch.ev_charger

Why This Works Better

  • No HACS dependency: Uses only built-in HA components (command_line, automation).
  • Can't crash HA: External API call. If it fails, sensor goes unavailable — nothing else breaks.
  • Cheapest window built-in: No separate Jinja macro, no template debugging, no DST bugs.
  • Cost estimate included: Know what you'll pay, not just when to run.
  • No API key needed: Free tier gives you 50 requests/day — enough for 3 sensors updating every 30 minutes.
  • Always current data: Prices update hourly from EPEX Spot. Next-day prices available from 14:00 CET.

Rate Limits and Fair Use

The free tier allows 50 requests per day per IP. Three sensors updating every 30 minutes = 144 requests/day, which exceeds the limit. Here's how to handle it:

  • Set scan_interval: 3600 (hourly) instead of 1800 — 3 sensors × 24 hours = 72 requests. Still over, but close.
  • Better: consolidate to 2 sensors with scan_interval: 1800 = 96 requests. Use the get_price_stats sensor (it includes min/max hours).
  • Best: get a Pro API key — 10,000 requests/day for production use.

Available Tools

Beyond the sensors shown above, Energy MCP provides 11 tools you can query from HA:

  • get_spot_prices — hourly EPEX Spot prices (up to 48h ahead)
  • get_price_forecast — tomorrow's day-ahead prices
  • get_price_stats — average, min, max, cheapest/expensive hours
  • find_cheapest_window — optimal time window + cost estimate
  • compare_tariffs — dynamic vs. fixed tariff comparison
  • calculate_energy_cost — cost of X kWh at current prices
  • analyze_price_trends — weekly trends and volatility
  • search_energy_prices — spot prices for 25+ EU countries
  • get_generation_mix — real-time solar/wind/gas/nuclear breakdown
  • get_carbon_intensity — gCO₂/kWh from generation mix
  • get_gas_prices — TTF Dutch benchmark gas prices
